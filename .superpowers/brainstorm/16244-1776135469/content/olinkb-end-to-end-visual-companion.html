1. Del wheel al workspace

El sistema empieza fuera del editor. Primero se instala el comando local, luego se prepara PostgreSQL, después se registran miembros y al final se escribe la configuración MCP que VS Code podrá lanzar.

Instala un comando, no un daemon
Instalación

Instala el CLI

El entrypoint publicado es olinkb. Puede llegar por release con pipx o por editable install desde el repo.

pipx install https://github.com/rzjulio/olinkb/releases/download/v0.1.0/olinkb-0.1.0-py3-none-any.whl
Infraestructura

Levanta PostgreSQL

La base no es opcional. OlinKB usa PostgreSQL como fuente de verdad y requiere URL de conexión para operar.

docker compose -f docker/docker-compose.yml up -d
OLINKB_PG_URL
OLINKB_TEAM
OLINKB_USER
Esquema

Aplica migraciones

Antes de cualquier sesión, el esquema debe existir: memorias, sesiones, auditoría, membresías y tablas auxiliares.

olinkb migrate
Identidad

Registra miembros

El sistema necesita saber quién eres y con qué rol trabajas. El rol gobierna qué puedes guardar, revisar o publicar.

olinkb add-member --username rzjulio --role admin
Editor

Bootstrap del workspace

olinkb --init escribe la configuración MCP y deja preparado olinkb-viewer/ para la experiencia visual.

olinkb --init

2. Qué hace realmente el bootstrap

El bootstrap no inicia el servidor todavía. Su trabajo es cablear el workspace y escribir los artefactos que luego permitirán a VS Code lanzar el MCP correctamente.

Lo visible y lo invisible

Lo que sí escribe

Dependiendo de si eliges configuración global o por repositorio, OlinKB deja el workspace listo para que el editor conozca el servidor.

Artefactos concretos
  • .vscode/mcp.json con el comando olinkb mcp y variables de entorno.
  • .github/copilot-instructions.md cuando el bootstrap es por repositorio.
  • olinkb-viewer/ con scaffold inicial para la experiencia visual.
  • Configuración global de VS Code si se eligió modo global.
repository scope
global scope
workspace auto-detect

Lo que todavía no hace

Bootstrap no valida que todo el runtime funcione extremo a extremo; solo deja listos los puntos de entrada.

No confundir
  • No mantiene un proceso vivo en background.
  • No hace login del viewer.
  • No crea sesiones de trabajo todavía.
  • No precarga memoria por sí mismo.
Resultado mental correcto: bootstrap_workspace() prepara el camino; el trabajo real empieza cuando VS Code o el navegador lanzan una superficie.
Before → After

Cómo cambia el workspace

Antes

El repo solo tiene código, docs, tests y un viewer estático. VS Code no sabe que existe un servidor MCP llamado OlinKB.

Después

El editor ya conoce un servidor local ejecutable, con entorno, equipo y proyecto listos para empezar a intercambiar memoria.

Variables efectivas

El runtime nace desde configuración

  • OLINKB_PG_URL: conexión a PostgreSQL.
  • OLINKB_TEAM: equipo del usuario.
  • OLINKB_USER: identidad por defecto.
  • OLINKB_PROJECT: proyecto por defecto si aplica.
  • OLINKB_PG_POOL_MAX_SIZE: tamaño del pool.

3. Cómo empieza a trabajar el MCP

Cuando VS Code necesita memoria, no llama a una API remota. Lanza un proceso local con olinkb mcp, registra tools FastMCP y delega toda la lógica real a OlinKBApp.

De stdio a PostgreSQL
1

El editor lee el mcp.json y entiende que hay un comando local listo para ser invocado.

2

Se lanza olinkb mcp

El proceso es local y efímero. No hay daemon residente que mantenga estado global.

3

FastMCP registra tools

El servidor expone hoy 8 tools públicas: boot, recall, save, proposal, review, end y forget.

4

App resuelve el trabajo

OlinKBApp valida permisos, usa caché, toca sesiones y delega SQL a PostgresStorage.

5

La respuesta vuelve al agente

El resultado viaja de vuelta por el protocolo MCP y se convierte en contexto utilizable en el chat.

4. Boot Session: de infraestructura a contexto útil

boot_session es el puente entre el runtime y la experiencia real del usuario. Aquí se abre la sesión activa, se resuelve membresía y se carga la primera porción de memoria relevante.

El boot no adivina: carga por prioridad

Prioridad de carga real

El SQL actual prioriza por prefijo URI y recencia. No filtra por memory_type antes de ordenar.

Memorias asociadas al proyecto activo, sin distinción por tipo.

4
personal://<user>/%

Contexto personal del desarrollador.

5. Buscar contexto y persistir conocimiento

La operación diaria se mueve entre dos verbos distintos: remember para recuperar y save_memory para consolidar conocimiento durable.

Buscar no es guardar

remember

Entrega memoria relevante bajo demanda a partir de query, scope, identidad y caché.

Recall
  • Resuelve contexto desde session_id o defaults.
  • Construye cache key por query, scope, usuario y proyecto.
  • Consulta caché local.
  • Si falla, usa pg_trgm sobre título, contenido, URI, tags, tipo y metadata.
  • Mezcla memorias con resúmenes de sesión y toca retrieval_count.
trigram similarity
tags enriquecidos
metadata::text
session summaries

save_memory

Es el punto único de escritura: inserta o actualiza según el URI, audita y limpia caché.

Persist
  • Valida scope, URI, memory_type y permisos.
  • Normaliza metadata y enriquece tags.
  • Calcula content_hash SHA256.
  • Crea o actualiza en memories.
  • Inserta registro en audit_log.
  • Invalida cachés boot:* y remember:*.

6. Promoción y documentación gestionada no son el mismo flujo

Este es el punto que más fácilmente se malinterpreta. Una memoria propuesta para revisión no equivale a una memoria de documentación técnica o de negocio.

Separación crucial
Regla mental correcta: propose_memory_promotion sirve para enviar una memoria existente de proyecto a revisión hacia convención. La documentación técnica o de negocio se crea hoy con save_memory usando tipos especiales o desde el viewer HTTP autenticado.

Flujo A · Promover una memoria existente

Sirve para revisión y aprobación, no para publicar documentación técnica.

Proposal → Review
1
Guardar memoria base
La memoria ya existe en scope project.
2
Proponer promoción
Se marca como pendiente con rationale.
3
Lead/Admin revisa
La cola llega por review queue y tools de aprobación.
4
Resultado
Si se aprueba, hoy converge en convention.

Flujo B · Crear documentación gestionada

Sirve para publicar documentation, business_documentation o development_standard.

Managed memory
1
Elegir tipo especial
documentation, business_documentation o development_standard.
2
Adjuntar metadata útil
documentation_scope, applicable_projects, authoring_surface, file_name.
3
Guardar con save_memory
Directamente desde MCP o a través del viewer.
4
Recuperación posterior
Se descubre por tags enriquecidos, metadata y búsqueda natural.

7. Viewer: lectura pública, autoría restringida

El viewer HTTP existe para hacer visible el sistema y para ofrecer una superficie de publicación mucho más guiada. No reemplaza al MCP; lo complementa.

Login para autoría, lectura para todos

Secuencia operativa del viewer

El backend HTTP hace más que recibir un POST. También autentica, normaliza y decide el scope real a persistir.

Browser → Viewer → App → PG
1. GET público
Cualquiera puede abrir el viewer y explorar la memoria disponible.
2. POST /api/auth/login
Se crea una sesión del viewer y se habilita el composer si el rol es lead o admin.
3. POST /api/memories
Se valida tipo, scope objetivo, archivo Markdown y permisos.
4. Normalización
global → scope org/shared, repo → scope project cuando aplica o metadata multi-repo.
5. save_memory
La creación real sigue pasando por OlinKBApp y PostgreSQL.

Tipos que el viewer sí expone hoy

La UI visual actual está pensada para docs curadas y no muestra todos los tipos posibles del dominio.

Permitidos
  • documentation: documentación técnica o de ingeniería.
  • business_documentation: documentación de negocio, solo para admins.

Tipo que no tiene UI dedicada

Existe en el dominio, pero la experiencia visual actual no lo crea explícitamente desde el composer.

No expuesto en UI
  • development_standard: válido en el sistema, pero sin formulario visual propio en el viewer actual.

8. Quién puede hacer qué

La autorización vive en la capa de aplicación y cambia según scope, tipo de memoria y superficie usada. La matriz visual evita mezclar reglas del MCP con reglas del viewer.

Matiz importante

Las reglas del viewer son más curadas que las del MCP genérico. El viewer está pensado como superficie de autoría aprobadora.

No son idénticas

Un developer puede guardar documentation de proyecto por MCP si tiene permisos de escritura de proyecto. Ese mismo developer no puede abrir el composer del viewer para publicar desde la UI visual actual.

Regla más sensible

La documentación de negocio se mantiene más restringida que el resto del conocimiento gestionado.

Admin only

Tanto en la aplicación como en el viewer, business_documentation solo puede ser guardada por un admin.

Gap

El sistema ya es útil, pero aún no es un retrieval engine semántico de siguiente nivel.

Future work

No hay embeddings ni deduplicación semántica. La búsqueda actual combina trigram, tags y metadata textual.

10. Ruta correcta de uso hoy

Si el usuario quiere entender rápidamente cómo debe usar OlinKB en el estado actual, esta es la secuencia mental y operativa más fiel al código.

Mapa final

Trabajo diario con agentes

  • boot_session abre contexto inicial y sesión activa.
  • remember recupera contexto adicional cuando la conversación lo necesita.
  • save_memory fija decisiones, bugs, procedimientos y hallazgos.
  • end_session cierra y resume el trabajo.

Gobierno y revisión

  • propose_memory_promotion envía una memoria a review.
  • list_pending_approvals muestra la cola para leads y admins.
  • review_memory_proposal aprueba o rechaza la propuesta.

Documentación visual y curada

  • olinkb viewer abre la superficie visual.
  • login del viewer habilita autoría para lead/admin.
  • composer publica documentación técnica o de negocio desde Markdown.

Si quieres explicar OlinKB a un usuario nuevo

Empieza por esta secuencia, no por las tablas.

Narrativa recomendada
  1. Instala y configura el comando local.
  2. Explica que VS Code lanza el MCP por stdio.
  3. Muestra que boot_session abre el contexto inicial.
  4. Distingue recall de escritura.
  5. Separa promoción de documentación gestionada.
  6. Cierra mostrando viewer, roles y gaps reales.

Resumen final. OlinKB ya funciona como un sistema operativo de memoria para agentes: instala un comando local, prepara el workspace, arranca un servidor MCP bajo demanda, crea contexto inicial con boot_session, recupera conocimiento con remember, persiste memoria durable con save_memory y, cuando hace falta una experiencia más guiada, abre un viewer visual para publicar documentación técnica o de negocio. La clave para entenderlo bien es no mezclar superficies, no mezclar promoción con documentación y no describir como “implementado” lo que hoy sigue siendo una capacidad parcial o futura.
