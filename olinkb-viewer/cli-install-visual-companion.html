Visual Companion · OlinKB CLI + MCP

Qué instala la CLI y qué pasa realmente cuando el editor usa OlinKB

Esta vista separa la historia en tres capas: instalación del paquete, bootstrap del workspace y ejecución MCP en tiempo real. La meta es que puedas ver, casi como una radiografía, qué aparece en la máquina del developer y qué se mueve detrás del telón.

Resumen en una frase

OlinKB no instala un daemon: instala un comando local olinkb, escribe configuración MCP para VS Code, usa PostgreSQL como memoria persistente y deja que el editor levante un proceso olinkb mcp por demanda vía stdio.

Timeline de instalación

De izquierda a derecha: lo que ejecuta el developer y la consecuencia material de cada paso.

Paquete

pipx instala el wheel

Se descarga o abre el wheel, se crea un venv aislado y se expone el launcher olinkb.

  • Script entrypoint: olinkb = olinkb.cli:main
  • Dependencias: mcp, asyncpg
  • No toca todavía VS Code ni PostgreSQL
Bootstrap

olinkb --init

Pregunta scope, URL de PostgreSQL y team. Detecta el nombre del proyecto si el scope es de repositorio.

  • Scope: repository o global
  • Construye el bloque MCP con env vars
  • Hace merge en lugar de reescribir ciegamente
Editor

VS Code queda cableado

El editor ahora sabe que existe un servidor MCP llamado olinkb y cómo arrancarlo.

  • Usa command: olinkb
  • Usa args: ["mcp"]
  • Pasa OLINKB_PG_URL, OLINKB_TEAM, OLINKB_USER
Base de datos

olinkb migrate

Se crean extensiones, tablas e índices. Aquí nace el storage persistente real.

  • Extensiones: uuid-ossp, pg_trgm
  • Tablas: team_members, memories, sessions, audit_log
  • La DB es la memoria compartida real
Uso real

El editor levanta el subprocess MCP

Cada vez que Copilot necesita memoria, puede arrancar olinkb mcp y hablar por stdin/stdout.

  • No hay puerto HTTP para MCP
  • No hay servicio siempre encendido
  • La caché vive solo mientras vive el proceso

Qué instala exactamente el wheel

Esta es la parte que normalmente queda invisible: el wheel no instala solo un comando, instala un paquete Python completo y un launcher que apunta al entrypoint del proyecto.

Lo que queda en disco

Tras pipx install olinkb-0.1.0-py3-none-any.whl, pipx crea un venv aislado y dentro instala el paquete Python con su código y metadatos.

site-packages/ olinkb/ __init__.py cli.py app.py server.py bootstrap.py olinkb-0.1.0.dist-info/
  • Sí, hay archivos .py reales instalados en tu máquina.
  • También quedan las dependencias mcp y asyncpg en ese mismo entorno.
  • El código no vive en tu Python global, sino en el entorno aislado que maneja pipx.

Cómo nace el comando olinkb

El wheel trae metadata para que el instalador genere un launcher a partir del script declarado en el proyecto.

[project.scripts] olinkb = "olinkb.cli:main"
  • Eso le dice al instalador qué función debe ejecutar el comando.
  • El launcher suele ser un script pequeño o shim que llama al Python del venv.
  • Cuando escribes olinkb ..., en realidad terminas ejecutando main() de olinkb/cli.py.
1. Shell Tu terminal resuelve olinkb desde el PATH.
2. Launcher El wrapper apunta al Python del venv creado por pipx.
3. Import Python importa olinkb.cli desde los archivos instalados.
4. Entry point Se ejecuta olinkb.cli:main y de ahí salen mcp, migrate, --init y demás.

Lo importante

Instalar el wheel sí deja software Python local en tu máquina. No es un acceso remoto al repo ni un alias vacío.

Lo que no hace

No instala PostgreSQL, no crea un daemon y no migra la base automáticamente. Solo deja el software local listo para usarse.

Lo visible y lo invisible

Esta separación aclara por qué OlinKB puede sentirse “simple” por fuera y, aun así, tener varias capas internas.

Lo que el developer sí ve

front stage
  • El comando olinkb aparece disponible en la terminal.
  • olinkb --init pide pocos datos y escribe config.
  • Se crean archivos en .vscode, .github y opcionalmente en el viewer scaffold.
  • olinkb migrate imprime migraciones aplicadas.
  • Luego el agente usa memoria sin pedir comandos manuales en cada prompt.

Lo que ocurre por detrás

back stage
  • El entrypoint del paquete entra por olinkb.cli:main.
  • El bootstrap hace merge de JSON e inserta o actualiza el protocolo en instrucciones del repo.
  • El runtime MCP crea un OlinKBApp con settings, cache TTL y session manager.
  • Las búsquedas remember usan trigram similarity con pg_trgm.
  • La persistencia real no vive en el proceso MCP: vive en PostgreSQL.

Mapa de la máquina del developer

Cuatro zonas: package install, editor config, subprocess MCP y base persistente.

1 · Python tooling

pipx / venv aislado

Es donde vive el paquete instalado y sus dependencias Python.

  • Wheel de OlinKB
  • Launcher olinkb
  • mcp y asyncpg
2 · Workspace o usuario

VS Code MCP registry

Es donde el editor aprende cómo arrancar el servidor y con qué variables de entorno.

  • .vscode/mcp.json o user mcp.json
  • Instructions del repo para activar las tools
  • Project name inyectado en modo repository
3 · Proceso efímero

Subprocess olinkb mcp

Vive mientras la conversación necesita el servidor MCP. Después desaparece.

  • Transport: stdin/stdout
  • Official MCP low-level router
  • ReadCache + SessionManager en RAM
4 · Persistencia

PostgreSQL compartido

Es la fuente de verdad que sobrevive a reinicios, cierres del editor y múltiples developers.

  • memories
  • sessions
  • audit_log
  • team_members

Archivos y artefactos que aparecen

No todo se crea en el mismo momento. Algunos artefactos son de bootstrap, otros son de migración y otros son puramente runtime.

Instalación del comando

Esto aparece por instalar el wheel con pipx.

pipx venv + launcher olinkb
  • Expone el entrypoint CLI
  • Contiene el código Python del paquete
  • No crea todavía schema ni config de VS Code

Bootstrap repository

Esto aparece cuando el scope es del repo actual.

.vscode/mcp.json .github/copilot-instructions.md olinkb-viewer/index.html olinkb-viewer/README.md
  • Registra el servidor MCP
  • Le enseña al agente cuándo usar memoria
  • Evita duplicar bloques al re-ejecutarse

Bootstrap global

Esto aparece cuando el developer elige scope global.

macOS ~/Library/Application Support/Code/User/mcp.json
  • Configura VS Code a nivel usuario
  • No escribe instrucciones del repo
  • Útil si no quieres commitear la integración

Schema PostgreSQL

Esto nace cuando se ejecuta la migración.

schema_migrations team_members memories sessions audit_log
  • También crea índices trigram GIN
  • Las extensiones quedan habilitadas en la DB
  • Aquí vive la memoria compartida

Estado efímero

Esto existe solo mientras vive el proceso MCP actual.

ReadCache (TTL) SessionManager settings cargados en RAM
  • No sobrevive al cierre del subprocess
  • Sirve para velocidad y contexto inmediato
  • No reemplaza a PostgreSQL

Integración automática

Lo más importante no es solo el binario; es la combinación de binario + registro MCP + instrucciones.

command: olinkb args: ["mcp"] env: OLINKB_*
  • Sin esto, el editor no lo invoca automáticamente
  • Sin instrucciones, el modelo puede usarlo de forma inconsistente
  • Sin DB, el servidor no tiene memoria real

Runtime MCP, paso por paso

Cambia de pestaña para seguir el recorrido interno del subprocess desde que VS Code lo levanta hasta que termina.

El editor ejecuta olinkb mcp

VS Code no llama funciones Python directamente. Ejecuta un proceso del sistema operativo y habla con él por stdio. Esa es la frontera MCP real.

Visible

No suele verse nada en pantalla. Solo “funciona” cuando el agente decide usar memoria.

Interno

El entrypoint CLI reconoce mcp o serve y delega a run_server().

Se leen variables de entorno

El proceso construye Settings a partir de OLINKB_PG_URL, OLINKB_TEAM, OLINKB_USER y opcionalmente OLINKB_PROJECT.

ObligatorioOLINKB_PG_URL y OLINKB_TEAM
FallbackSi no existe OLINKB_USER, usa USER o USERNAME.
OpcionalProject por defecto y parámetros del cache TTL.

El SDK MCP publica las tools

El servidor low-level del SDK oficial registra funciones como boot_session, remember, save_memory, end_session y forget.

boot_session

Arranca sesión y precarga memoria relevante.

remember

Busca memorias por trigram similarity y puede devolver payload lean.

save_memory

Persiste memoria con deduplicación por hash y metadata opcional.

La tool entra en OlinKBApp

Aquí se conectan reglas de negocio, caché y acceso a base. Este es el cerebro operativo del subprocess.

ReadCache

Evita repetir lecturas costosas en el mismo proceso. TTL por defecto: 300 segundos.

SessionManager

Guarda estado de la sesión activa solo en memoria del proceso.

Ejemplo mental

remember intenta cache, si falla consulta PostgreSQL, mezcla resultados, toca contadores y devuelve contexto al agente.

La DB hace el trabajo persistente

PostgreSQL no es un accesorio. Es la fuente de verdad del sistema: membresía, memorias, sesiones y trazabilidad.

pg_trgm

Permite similarity y matching difuso para remember.

audit_log

Guarda trail de acciones y forgets. Sirve para trazabilidad.

sessions

Permite cerrar sesiones y persistir summaries más allá del proceso actual.

La conexión termina y el proceso muere

Al cerrarse la conexión MCP, el subprocess termina. Lo que estaba en cache o en session manager desaparece. Lo que estaba en PostgreSQL queda.

Se conserva

+
Memories

Persisten en la tabla correspondiente.

+
Audit trail

Queda en audit_log.

+
Sessions cerradas

Sobreviven al reinicio del editor.

Se pierde

−
ReadCache

Es solo RAM del proceso actual.

−
SessionManager vivo

No sobrevive al cierre del subprocess.

−
Estado efímero del boot

Se recalcula al arrancar un nuevo proceso.

Qué sí se instala y qué no

Este es el punto que más confunde: instalar la CLI no equivale a instalar toda la plataforma alrededor.

Lo que OlinKB sí deja listo

+
CLI local

El comando olinkb queda utilizable desde terminal y por VS Code.

+
Dependencias Python

mcp para protocolo MCP y asyncpg para hablar con PostgreSQL.

+
Registro MCP

La integración con VS Code queda configurada por archivo JSON.

+
Protocolo del repo

El agente aprende cuándo debe usar memoria automáticamente.

Lo que el developer debe tener o provisionar

−
Python 3.11+

Precondición del paquete, no parte de OlinKB.

−
pipx

Herramienta de instalación recomendada, externa a OlinKB.

−
PostgreSQL corriendo

Puede ser local con Docker o remoto; OlinKB no te lo provisiona por sí mismo.

−
VS Code + Copilot

El cliente que invoca MCP ya debe existir y estar operativo.

Resumen operativo

Si tuvieras que contárselo a otro developer en 20 segundos, esta sería la versión correcta.

OlinKB instala un comando local, no un servicio residente. Ese comando registra un servidor MCP en VS Code, usa variables de entorno para saber a qué PostgreSQL conectarse, y el editor lo arranca como subprocess por demanda. PostgreSQL guarda la memoria duradera; el proceso MCP solo aporta routing, caché efímera y coordinación de tools.

Tip de lectura Si quieres recorrerlo como onboarding: primero mira Timeline, luego Mapa de máquina y finalmente Runtime MCP.
