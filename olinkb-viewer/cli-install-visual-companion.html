What the CLI installs and what really happens when the editor uses OlinKB

This view separates the story into three layers: package installation, workspace bootstrap, and real-time MCP execution. The goal is to let you see, almost like an X-ray, what appears on the developer's machine and what moves behind the curtain.

Summary in one sentence

OlinKB does not install a daemon: it installs a local olinkb command, writes MCP configuration for VS Code, uses PostgreSQL as persistent memory, and lets the editor launch an olinkb mcp process on demand over stdio.

Installation timeline

From left to right: what the developer runs and the material consequence of each step.

Paquete

pipx installs the wheel

The wheel is downloaded or opened, an isolated venv is created, and the olinkb launcher is exposed.

  • Script entrypoint: olinkb = olinkb.cli:main
  • Dependencies: mcp, asyncpg
  • It does not touch VS Code or PostgreSQL yet
Bootstrap

olinkb --init

It asks for scope, the PostgreSQL URL, and the team. It detects the project name if the scope is repository-level.

  • Scope: repository o global
  • Builds the MCP block with env vars
  • Merges instead of blindly overwriting
Editor

VS Code gets wired up

The editor now knows that an MCP server called olinkb exists and how to start it.

  • Usa command: olinkb
  • Usa args: ["mcp"]
  • Passes OLINKB_PG_URL, OLINKB_TEAM, OLINKB_USER
Database

olinkb migrate

Extensions, tables, and indexes are created. This is where real persistent storage begins.

  • Extensiones: uuid-ossp, pg_trgm
  • Tablas: team_members, memories, sessions, audit_log
  • The DB is the real shared memory
Real use

The editor launches the MCP subprocess

Whenever Copilot needs memory, it can start olinkb mcp and talk over stdin/stdout.

  • There is no HTTP port for MCP
  • There is no always-on service
  • The cache lives only as long as the process lives

What the wheel actually installs

This is the part that usually stays invisible: the wheel does not install only a command, it installs a full Python package and a launcher that points to the project's entrypoint.

What lands on disk

After pipx install olinkb-0.1.0-py3-none-any.whl, pipx creates an isolated venv and installs the Python package with its code and metadata inside it.

site-packages/ olinkb/ __init__.py cli.py app.py server.py bootstrap.py olinkb-0.1.0.dist-info/
  • Yes, real .py files are installed on your machine.
  • The mcp and asyncpg dependencies also live in that same environment.
  • The code does not live in your global Python, but in the isolated environment managed by pipx.

How the olinkb command is created

The wheel carries metadata so the installer can generate a launcher from the script declared in the project.

[project.scripts] olinkb = "olinkb.cli:main"
  • That tells the installer which function the command must execute.
  • The launcher is usually a small script or shim that calls the venv's Python.
  • When you type olinkb ..., you are really executing main() from olinkb/cli.py.
1. Shell Your terminal resolves olinkb from the PATH.
2. Launcher The wrapper points to the Python in the venv created by pipx.
3. Import Python imports olinkb.cli from the installed files.
4. Entry point olinkb.cli:main runs, and from there mcp, migrate, --init, and the rest branch off.

What matters

Installing the wheel does leave local Python software on your machine. It is not remote access to the repo or an empty alias.

What it does not do

It does not install PostgreSQL, does not create a daemon, and does not migrate the database automatically. It only leaves the local software ready to use.

The visible and the invisible

This separation clarifies why OlinKB can feel simple from the outside and still have multiple internal layers.

What the developer does see

front stage
  • The olinkb command appears in the terminal.
  • olinkb --init asks for a small amount of data and writes config.
  • Files are created in .vscode, .github, and optionally in the viewer scaffold.
  • olinkb migrate prints the applied migrations.
  • After that, the agent uses memory without asking for manual commands in every prompt.

What happens behind the scenes

back stage
  • The package entrypoint comes in through olinkb.cli:main.
  • Bootstrap merges JSON and inserts or updates the protocol in the repo instructions.
  • The MCP runtime creates an OlinKBApp with settings, cache TTL, and a session manager.
  • remember searches use trigram similarity with pg_trgm.
  • Real persistence does not live in the MCP process: it lives in PostgreSQL.

Schema PostgreSQL

This is created when the migration runs.

Ephemeral state

This exists only while the current MCP process is alive.

  • Does not survive subprocess shutdown
  • Helps with speed and immediate context

Automatic integration

The most important part is not only the binary; it is the combination of binary + MCP registry + instructions.

command: olinkb args: ["mcp"] env: OLINKB_*
  • Without this, the editor does not invoke it automatically
  • Without instructions, the model may use it inconsistently
  • Without the DB, the server has no real memory

MCP runtime, step by step

Switch tabs to follow the subprocess's internal path from the moment VS Code launches it until it ends.

The editor executes olinkb mcp

VS Code does not call Python functions directly. It executes an operating-system process and talks to it over stdio. That is the real MCP boundary.

Visible

Nothing usually appears on screen. It simply "works" when the agent decides to use memory.

Internal

The CLI entrypoint recognizes mcp or serve and delegates to run_server().

Environment variables are read

The process builds Settings from OLINKB_PG_URL, OLINKB_TEAM, OLINKB_USER, and optionally OLINKB_PROJECT.

RequiredOLINKB_PG_URL y OLINKB_TEAM
FallbackSi no existe OLINKB_USER, it uses USER or USERNAME.
OptionalDefault project and cache TTL parameters.

The MCP SDK publishes the tools

The low-level server from the official SDK registers functions such as boot_session, remember, save_memory, end_session, and forget.

boot_session

Starts a session and preloads relevant memory.

remember

Searches memories by trigram similarity and can return a lean payload.

save_memory

Persists memory with hash-based deduplication and optional metadata.

The tool enters OlinKBApp

This is where business rules, cache, and database access come together. This is the subprocess's operational brain.

ReadCache

Avoids repeating expensive reads in the same process. Default TTL: 300 seconds.

SessionManager

Keeps active-session state only in process memory.

Mental model

remember tries the cache, falls back to PostgreSQL on a miss, combines results, updates counters, and returns context to the agent.

The DB does the persistent work

PostgreSQL is not an accessory. It is the system's source of truth: membership, memories, sessions, and traceability.

pg_trgm

Enables similarity and fuzzy matching for remember.

audit_log

Stores an action trail and forgets. Used for traceability.

sessions

Makes it possible to close sessions and persist summaries beyond the current process.

The connection ends and the process dies

When the MCP connection closes, the subprocess ends. Whatever lived in cache or the session manager disappears. Whatever lived in PostgreSQL remains.

Retained

+
Memories

Persist in the corresponding table.

+
Audit trail

Remains in audit_log.

+
Closed sessions

Survive editor restarts.

Lost

−
ReadCache

It is only RAM in the current process.

−
Live SessionManager

Does not survive subprocess shutdown.

−
Ephemeral boot state

Is recalculated when a new process starts.

Operational summary

If you had to explain it to another developer in 20 seconds, this would be the correct version.

OlinKB installs a local command, not a resident service. That command registers an MCP server in VS Code, uses environment variables to know which PostgreSQL instance to connect to, and the editor starts it as an on-demand subprocess. PostgreSQL stores durable memory; the MCP process only contributes routing, ephemeral cache, and tool coordination.
